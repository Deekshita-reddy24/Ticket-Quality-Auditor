#  AI Usage Note & Architecture Overview

**Project Name:** SD-13: AI Ticket Quality Auditor  
**Core LLM Engine:** Llama-3.3-70b (Hosted via Groq Cloud API)  
**Backend Framework:** FastAPI (Python)  
**Data Validation Layer:** Pydantic v2  
**Database Persistence:** SQLite3  

---

## 🤖 1. Model Details & Infrastructure
This prototype explicitly utilizes the state-of-the-art **Llama-3.3-70b** model, leveraging the ultra-low latency infrastructure provided by the **Groq Cloud API**. 
* **Model Selection Criteria**: Chosen for its high-performance 70-billion parameter reasoning capabilities, allowing it to perform deep contextual analysis on technical support data instantly.
* **API Integration**: Connected securely using the Groq Python SDK, utilizing asynchronous worker environments to process ticket payloads without blocking the main event loop.

---

## 📝 2. Prompt Engineering Strategy

We implemented advanced, custom system prompts to instruct the LLM to act as an automated ticket quality auditor. The core engineering strategies include:

* **Role-Based Instruction**: The system prompt forces Llama-3.3-70b to adopt the persona of a *Senior QA Engineer and Technical Support Lead*.
* **Multi-Dimensional Scoring Rubric**: Instructs the model to evaluate incoming support tickets based on three strict operational parameters:
  1. **Clarity**: Is the issue understandable without ambiguity?
  2. **Tone**: Is the language professional, constructive, and objective?
  3. **Resolution Accuracy**: Does it provide enough context (like steps to reproduce) to allow an engineer to fix it immediately?
* **Structured JSON Outputs**: To prevent conversational fluff from breaking our code parser, the prompt mandates that the model output its analysis *exclusively* in a rigid, predefined JSON structure.

## 🛠️ 3. End-to-End System Integration

AI assistance was utilized as a technical co-pilot to construct the scaffolding for the entire runtime pipeline:

### A. Data Validation (Pydantic Models)
* Formulated the Python data structures to strictly validate incoming user inputs and outgoing JSON analysis payloads generated by Groq.
* Managed fields for ticket components like titles, descriptions, and structural arrays.

### B. Multi-Turn Clarification Agent Loop
* Assisted in writing the stateful backend logic. If the Llama model calculates a quality score below the 80-point threshold, the platform activates a conversational multi-turn loop.
* The chatbot leverages Groq's high-speed inference to ask up to 3 highly targeted questions to gather missing details from the user.

### C. Relational Data Persistence (SQLite3)
* Designed a clean, relational schema to log all audited ticket histories inside a local `tickets.db` database.
* Every entry tracks the ticket details, computed quality scores, conversational history logs, and the final routing assignment queue (Backend, Frontend, or Database Team).
